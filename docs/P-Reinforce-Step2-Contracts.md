# P-Reinforce Step 2 Contracts

## Why Step 2 Exists

Step 2 exists to prevent a beautiful interface from becoming a brittle knowledge toy.

At this stage, the goal is to define:

- what the system stores
- what the system derives
- what can be trusted as canonical
- what can be regenerated later

If this is done poorly, the user can lose years of structure to early implementation mistakes.
If this is done well, the system can evolve across models, UIs, and workflows without losing its spine.

## Current Step

Current roadmap step:

`Step 2. Persistent Knowledge Contracts`

This step should answer six questions:

1. What is a raw source?
2. What is a wiki node?
3. What metadata is mandatory?
4. Which artifacts are canonical?
5. Which artifacts are derived?
6. How are identity and provenance preserved across years?

## Canonical vs Derived

This distinction is a foundation rule.

### Canonical Artifacts

These are authoritative.

| Artifact | Why canonical |
|---|---|
| `00_Raw/.../manifest.json` | describes immutable evidence |
| `00_Raw/.../source.md` and attachments | source of truth |
| `10_Wiki/**/*.md` | persistent synthesized knowledge |
| `20_Meta/policy.json` | machine-usable learned preference |
| `20_Meta/Policy.md` | human-readable reasoning for policy |
| `20_Meta/events/**/*.jsonl` | append-only semantic timeline |
| `20_Meta/AGENTS.md` | operational schema and behavior rules |

### Derived Artifacts

These are rebuildable.

| Artifact | Why derived |
|---|---|
| `20_Meta/index.md` | generated navigation surface |
| `20_Meta/index.json` | generated machine index |
| `20_Meta/graph.cache.json` | generated graph cache |
| `20_Meta/reports/*` | generated health and audit outputs |

Rule:

- canonical artifacts may produce derived artifacts
- derived artifacts must never become the only source of truth

## Identity Rules

### Source IDs

Pattern:

`src_YYYYMMDD_<random>`

Example:

`src_20260414_ab12cd34`

Properties:

- immutable
- stable across reprocessing
- referenced from wiki nodes and event logs

### Node IDs

Pattern:

`node_<type>_<stable-slug>`

Example:

`node_topics_persistent-wiki-compounding-artifact`

Properties:

- stable across title changes
- stable across folder moves
- the machine identity of the page

### Event IDs

Pattern:

`evt_YYYYMMDD_HHMMSS_suffix`

Example:

`evt_20260414_112045_a1b2`

Properties:

- append-only
- chronological
- collision-safe inside bursty pipelines
- easy to grep and audit

## Path Rules

### Raw Path Rule

Raw sources are date-bucketed for audit clarity:

`00_Raw/YYYY/MM/DD/source_<source_id>/`

Inside each source bundle:

- `manifest.json`
- `source.md`
- `attachments/`

### Wiki Path Rule

Wiki pages live under semantic top-level categories:

- `10_Wiki/Projects/`
- `10_Wiki/Topics/`
- `10_Wiki/Decisions/`
- `10_Wiki/Skills/`
- `10_Wiki/Views/`

Important:

- path is not identity
- frontmatter `id` is identity
- path is a semantic view

## Provenance Rule

Every active wiki node must reference at least one raw source through `source_refs`.

If a node is composed from multiple nodes instead of directly from raw sources, it still needs provenance.
That provenance can be traced through:

- direct `source_refs`
- related nodes with their own `source_refs`
- event history

No page should become persuasive without evidence.

## Frontmatter Rule

The wiki page frontmatter is the canonical machine contract for persistent markdown pages.

Mandatory fields:

- `id`
- `schema_version`
- `node_type`
- `title`
- `status`
- `confidence_score`
- `created_at`
- `updated_at`
- `last_reinforced`
- `category_path`
- `source_refs`
- `related`
- `contradicts`
- `policy_version`

Optional but recommended:

- `last_commit`
- `aliases`

## Linking Rule

There are two link layers.

### Human Reading Links

Inside markdown:

- use `[[Wiki Links]]`

Purpose:

- reading flow
- Obsidian-style browsing
- human comprehension

### Machine Relationship Links

Inside frontmatter and caches:

- use stable IDs

Purpose:

- survive renames
- survive folder moves
- power graph rebuilds

## Index Generation Rules

`20_Meta/index.json` should be generated by scanning all active wiki nodes.

For each page include:

- `node_id`
- `title`
- `node_type`
- `path`
- `status`
- `updated_at`
- `source_ref_count`
- optional one-line summary

`20_Meta/index.md` is the human-readable companion.

Generation rules:

1. ignore archived pages unless explicitly requested
2. sort by top-level category and title
3. one entry per node ID
4. if duplicate node IDs are found, fail generation and create an error event

## Graph Cache Generation Rules

`20_Meta/graph.cache.json` should be generated from:

- frontmatter `related`
- frontmatter `contradicts`
- category relationships
- inline `[[wikilink]]` references

Important:

- category relationships require derived category hub nodes such as `cat_topics`
- graph cache must support both wiki nodes and category nodes
- category hub nodes are rebuildable graph helpers, not canonical markdown pages

Edge relation types:

- `related`
- `contradicts`
- `parent_category`
- `wikilink`

Rules:

1. graph cache is never edited manually
2. duplicate edges should be merged
3. missing node targets should create lint warnings
4. graph cache should be safe to delete and rebuild

## Event Logging Rules

`20_Meta/events/` should be append-only.

Recommended storage:

`20_Meta/events/YYYY/MM/DD.jsonl`

Each line:

- one valid `EventLogEntry`
- one semantic operation

Examples of event types:

- `capture`
- `ingest_propose`
- `ingest_apply`
- `reinforce_update`
- `lint`
- `git_commit`
- `git_push`
- `migration`
- `rebuild`
- `error`

## What Became True After This Step

After these contracts exist, the project can now say:

- raw evidence has a formal identity
- wiki nodes have a formal identity
- policy has a formal shape
- semantic operations have a formal log entry shape
- index and graph outputs are explicitly treated as derived

That is a major shift from "an app that generates markdown" to "a system with durable knowledge contracts."

## What Is Still Not True

Even after Step 2 contracts are written, the system still does not yet:

- persist raw source bundles in production
- save real wiki node files from the UI
- write event logs automatically
- regenerate index and graph caches automatically
- run local watcher or Git automation

That work belongs to the next implementation slices.

## Immediate Follow-Up

The next implementation move should be:

1. wire the Studio UI to create `RawSourceManifest`
2. wire node generation to emit valid frontmatter
3. write events for propose/apply actions
4. materialize `index.json` and `graph.cache.json` from stored nodes

This is the shortest path from contracts to a real persistent wiki engine.
